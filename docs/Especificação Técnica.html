premissa central

O protótipo aprovado é a fonte da verdade visual. Esta spec traduz aquele HTML/JSX exploratório para uma base de produção sustentável — sem reinventar o visual, e mantendo o mesmo vocabulário (tecnomancia, cybermagia, ciclos lunares, "rituais").

01

Objetivo & princípios

por quê

Construir o site público da encodere.se e a área logada da aluna como um único projeto Astro, otimizado para conteúdo (marketing/currículo é estático) mas com ilhas React onde há interatividade real (filtros de trilha, dashboard, toggles).

Princípios de engenharia

  • Estático por padrão, JS sob demanda. Cada quilobyte de JavaScript precisa se justificar. Landing e trilhas renderizam como HTML puro; React só "hidrata" onde há estado.
  • O conteúdo é dado, não markup. Disciplinas, mestras, depoimentos e fases vivem em content collections tipadas — não hardcoded em JSX como no protótipo.
  • Tokens primeiro. Nenhuma cor literal em componente. Tudo via CSS custom properties já modeladas no protótipo (oklch violeta/cyan).
  • Acessível e legível no claro e no escuro. O cyberpunk é sutil — contraste AA é inegociável.
  • Direct-editável e tipado. TypeScript em tudo; componentes pequenos e single-purpose.

Não-objetivos (v1)

  • Não construir CMS próprio — conteúdo em arquivos versionados (MDX/JSON) basta para o volume atual.
  • Não construir LMS completo. O dashboard v1 é um painel de leitura + links; entregas e vídeo-aulas ficam em ferramentas externas (Discord, Zoom, repositório).
  • Não suportar i18n agora — só PT-BR. A estrutura deve permitir adicionar depois.

regra de contraste

O violeta escuro de marca (0.72 0.21 305) não tem contraste AA sobre o --encodere-bg escuro para texto pequeno. Use-o em texto ≥18px/bold, bordas e fills — nunca em corpo de texto. Para links/labels pequenos no escuro, suba a luminância para ~0.80.

tokens.css — forma

/* claro = base; escuro sobrescreve via [data-theme="dark"] */
:root {
  --encodere-bg:      oklch(0.97 0.005 295);
  --encodere-fg:      oklch(0.18 0.02 295);
  --encodere-violet:  oklch(0.45 0.22 305);
  --encodere-cyan:    oklch(0.55 0.14 200);
  /* …rule, fg-2, fg-3, bg-elev, bg-card… */
  --encodere-step: 4px;            /* escala de espaço base */
  --encodere-font: 'JetBrains Mono', ui-monospace, monospace;
}
[data-theme="dark"] {
  --encodere-bg:      oklch(0.13 0.02 295);
  --encodere-fg:      oklch(0.96 0.008 295);
  --encodere-violet:  oklch(0.72 0.21 305);
  --encodere-cyan:    oklch(0.84 0.14 195);
}

05

Modo claro / escuro

sem flash

O protótipo detecta prefers-color-scheme e persiste por artboard no localStorage. Em produção, unificamos num único tema global com script anti-flash inline no <head>, antes do paint.

  1. Script bloqueante no <head>: lê localStorage['encodere-theme']; se ausente, cai em matchMedia('(prefers-color-scheme: dark)'); aplica data-theme no <html> antes de qualquer render.
  2. Toggle (ilha React, client:idle): alterna data-theme e grava no localStorage. É o único JS de tema na página.
  3. CSS reage só via [data-theme="dark"] — nenhum componente sabe do tema, apenas consome tokens.
<!-- BaseLayout.astro · inline, antes do CSS -->
<script is:inline>
  const k = 'encodere-theme';
  const saved = localStorage.getItem(k);
  const sys = matchMedia('(prefers-color-scheme: dark)').matches;
  const theme = saved ?? (sys ? 'dark' : 'light');
  document.documentElement.dataset.theme = theme;
</script>
acessibilidade

Respeitar prefers-reduced-motion: glitch, scanline e boot só animam em @media (prefers-reduced-motion: no-preference). O estado final (conteúdo visível) é sempre o default — animações partem dele, nunca o escondem.

nota sobre GlitchTitle

O glitch é puramente CSS (pseudo-elementos ::before/::after + @keyframes, já no protótipo). Pode até virar .astro com classe utilitária e dispensar a ilha — só precisa do atributo data-text. Preferir essa versão sem-JS quando o título não muda em runtime.

Sobre as 3 variações de landing

O protótipo entregou três direções (terminal sutil, manifesto-zine, holo grid). Em produção:

  • Escolher uma como / canônica (recomendo a manifesto-zine — mais identidade, melhor pra topo de funil).
  • Manter as outras como variantes A/B sob /v/terminal e /v/holo, ou atrás de middleware de experimento, reaproveitando as mesmas seções (Roadmap, Depoimentos, CTA) com layouts diferentes.
  • As seções compartilhadas vivem em components/site/ e recebem o conteúdo por props — landing é só composição.
08

Modelagem de conteúdo

collections

No protótipo, disciplinas e depoimentos são arrays hardcoded dentro do JSX. Em produção viram content collections tipadas por Zod — editáveis sem tocar em componente.

content/config.ts

import { defineCollection, z } from 'astro:content';

const disciplinas = defineCollection({
  type: 'content',
  schema: z.object({
    fase:   z.enum(['I','II','III','IV','V']),
    nome:   z.string(),
    horas:  z.number(),
    mestra: z.string(),          // ref → mestras
    resumo: z.string(),
    tags:   z.array(z.string()),
    ordem:  z.number(),
  }),
});

const depoimentos = defineCollection({
  type: 'data',
  schema: z.object({
    nome: z.string(), turma: z.string(), papel: z.string(),
    texto: z.string(), projeto: z.string().optional(),
  }),
});

export const collections = { disciplinas, depoimentos, /* mestras, fases */ };

A página de trilhas então consome assim — totalmente data-driven:

---
import { getCollection } from 'astro:content';
const disciplinas = (await getCollection('disciplinas'))
  .sort((a,b) => a.data.ordem - b.data.ordem);
---
<CatalogoTrilhas client:visible disciplinas={disciplinas.map(d => d.data)} />
benefício

As mestras editam um .md por disciplina; o build valida o schema e quebra cedo se algo estiver errado. Nada de procurar string em JSX.

orçamento de animação

Nada de loop decorativo infinito em conteúdo (só o cursor, que é pequeno e semântico). Tudo gated em prefers-reduced-motion. Boot roda uma vez por sessão — não a cada navegação — pra não irritar.

Boot só uma vez · padrão

const [booting, setBooting] = useState(
  () => !sessionStorage.getItem('encodere-booted')
);
function finish() {
  sessionStorage.setItem('encodere-booted', '1');
  setBooting(false);
}

O glitch e a scanline já estão prontos como CSS no protótipo (bloco #encodere-keyframes em shared.jsx) — basta migrar para styles/global.css e referenciar por classe.

10

Dashboard & auth

área da aluna

A rota /aluna é SSR e protegida. Render no servidor já com os dados da aluna; React hidrata só as partes interativas (rituais que marcam, toggle de tema).

Fluxo de sessão

  1. Entrar: aluna informa e-mail em /aluna/entrar → recebe magic-link (combina com o tom "processo por carta").
  2. Callback: link valida token, cria sessão (cookie httpOnly, Lucia/Auth.js).
  3. Guarda: AlunaLayout.astro checa sessão no topo; sem ela → Astro.redirect('/aluna/entrar').
  4. Dados: progresso, próximo encontro, rituais e diário vêm de uma API/DB (v1 pode ser JSON por aluna + Turso/SQLite).
--- // src/pages/aluna/index.astro
import AlunaLayout from '../../layouts/AlunaLayout.astro';
const session = await getSession(Astro.request);
if (!session) return Astro.redirect('/aluna/entrar');
const aluna = await getPainel(session.userId);
---
<AlunaLayout aluna={aluna}>
  <ProgressoCiclo client:load fase={aluna.fase} pct={aluna.pct} />
  <ProximoEncontro encontro={aluna.proximo} />
  <RituaisPendentes client:visible rituais={aluna.rituais} />
  <DiarioFeiticos commits={aluna.diario} />
</AlunaLayout>
modelo de dados mínimo (v1)

Aluna { id, nome, turma, fase, pct } · Ritual { id, alunaId, titulo, prazo, feito } · EntradaDiario { id, alunaId, quando, texto, ref }. Turso/SQLite cobre de sobra o volume de uma turma.
